End-to-End Request Flow — Bankruptcy Shifting Process
1
User Login
Risk Manager authenticates via Keycloak SSO. → JWT Token issued
›
2
API Request
Frontend calls POST /api/v1/bankruptcy with Bearer JWT in Authorization header. → HTTPS to Backend
›
3
Token Validation
Spring Security validates JWT against Keycloak. Checks ROLE_risk-manager claim. → Access Granted
›
4
SaaS Token Request
Zeebe client calls login.cloud.camunda.io using client_credentials grant. → Access Token returned
›
5
Token Cached
Token stored at /tmp/.camunda/credentials. Reused for all calls until expiry. → No repeat auth calls
›
6
Process Started
gRPC call to Zeebe engine with Bearer token. BPMN process instance created. → fra-1.zeebe.camunda.io:443
›
7
Workflow Executes
Camunda SaaS executes Bankruptcy_workflow BPMN tasks. Visible in Operate. → Process Complete
